# Cortex Search Service の検索スコア指標解説

## 結果の例

```json
"@scores": {
    "cosine_similarity": 0.38239306,
    "reranker_score":   -6.875,
    "text_match":        0.057075117
}
```

Cortex Search Service の検索結果には3つのスコアが付与されます。
これらを組み合わせた `_relevance_score` が最終的な順位付けに使われます。

---

## 各指標の意味と判断基準

### 1. cosine_similarity（コサイン類似度）

**何を測っているか**

テキストをベクトル（数百次元の数値配列）に変換したとき、
クエリベクトルと文書ベクトルが「同じ方向を向いているか」を測る指標です。

統計検定２級で学ぶピアソン相関係数 `r` と数学的に同じ構造を持ちます。
相関係数が「偏差ベクトルの角度」を測るのに対し、
コサイン類似度は「ベクトルそのものの角度」を測ります（平均中心化しない版）。

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)
```

**値の範囲と目安**

| 値 | 目安 |
|----|------|
| 0.90 以上 | ほぼ同じ意味（株式会社NTTデータ ↔ NTTデータ株式会社）|
| 0.80〜0.90 | 高い類似性（NTTデータ ↔ NTT Data）|
| 0.60〜0.80 | 中程度の類似性（同業種・関連語）|
| 0.40〜0.60 | 弱い類似性（テーマが近いだけ）|
| 0.40 未満 | ほぼ無関係 |

今回の例 `0.38` は「ほぼ無関係」の領域です。
これは日立製作所のクエリに対して Amazon Japan がマッチした結果であり、
スコアの低さが「誤マッチ」の可能性を正しく示しています。

**注意点**

コサイン類似度は「方向」だけを見るため、ベクトルの大きさ（長さ）は無視されます。
短い単語（「NTT」など）と長い正式名称（「日本電信電話株式会社」）でも
意味が近ければ高スコアになります。これが表記揺れ吸収における強みです。

---

### 2. text_match（テキストマッチスコア）

**何を測っているか**

クエリ文字列とインデックス内テキストの「字面の一致度」を測る指標です。
BM25（Best Match 25）という確率的な文字列検索モデルに相当します。

BM25 は TF-IDF（単語頻度 × 逆文書頻度）を改良したもので、
統計的には「クエリ単語がどれだけレアで、かつどれだけ文書内に登場するか」
を確率モデルで表現しています。

**値の範囲と目安**

| 値 | 意味 |
|----|------|
| 0.5 以上 | 主要な単語が複数一致（非常に高いキーワード一致）|
| 0.1〜0.5 | 部分的な単語一致 |
| 0.1 未満 | ほぼ文字列一致なし |

今回の例 `0.057` はほぼゼロに近く、
「株式会社日立製作所」と「Amazon Japan合同会社」の間に
共通する単語が存在しないことを示しています。

**cosine_similarity との違い**

| 観点 | cosine_similarity | text_match |
|------|------------------|------------|
| 一致の種類 | 意味的な類似（semantics）| 字面の一致（lexical）|
| 「Toyota」→「トヨタ」| 高スコア（意味が同じ）| 低スコア（文字が違う）|
| 「NTT」→「NTTデータ」| 中スコア（部分的な意味）| 中スコア（文字が含まれる）|
| 活用場面 | 翻訳・言い換え・類義語 | スペルが分かっているとき |

表記揺れ（日英混在・略称）の名寄せには
`text_match` だけでは不十分で、`cosine_similarity` が主指標になります。

---

### 3. reranker_score（リランカースコア）

**何を測っているか**

初期検索（ベクトル検索 + テキストマッチ）で絞り込まれた候補を、
より精度の高いモデルで「再評価」したスコアです。

クロスエンコーダー（cross-encoder）と呼ばれる方式で、
クエリと文書を**同時に**モデルへ入力して関連性スコアを算出します。
ベクトル検索が「個別にベクトル化してから比較」するのに対し、
リランカーは「2つを並べてまとめて判断」します。

統計的な比喩として整理すると：
- ベクトル検索 = 各変数の単変量統計量（平均・分散）で個別に判断
- リランカー = 多変量解析で変数間の交互作用も含めて総合判断

**値の範囲と目安**

リランカースコアは正規化されておらず、任意の実数値を取ります（負値も多い）。
絶対値よりも**同一クエリ内での相対的な順位**が重要です。

| 相対的な位置 | 意味 |
|-------------|------|
| 候補内で最も大きい値 | 最も関連性が高い |
| 0 付近 | 関連性は中程度 |
| 大きな負値（例: -7, -10）| 関連性が低いと判断された |

今回の例 `-6.875` はかなり大きな負値であり、
リランカーも「このマッチは信頼できない」と判断しています。
cosine_similarity の `0.38` という低い値とも一致しており、
2つの指標が「非マッチ」という同じ結論を支持しています。

---

## 3指標をどう組み合わせて判断するか

Snowflake の `_relevance_score` は3指標の加重和（内部計算式は非公開）ですが、
手動で判断する場合は以下のフレームワークが使いやすいです。

```
判断ロジック:

cosine_similarity >= 0.80            → ほぼ確実にマッチ（英語表記・略称も含む）
cosine_similarity >= 0.60
  AND text_match  >= 0.10            → マッチ可能性が高い
cosine_similarity <  0.50
  AND reranker_score < -5            → 非マッチと判断
```

**具体例**

| クエリ → 候補 | cosine | text_match | reranker | 判断 |
|-------------|--------|------------|----------|------|
| NTTデータ → NTT Data | 0.88 | 0.31 | -1.2 | ✅ マッチ |
| Toyota → トヨタ自動車(株) | 0.82 | 0.04 | -2.0 | ✅ マッチ（意味一致） |
| 日立製作所 → Amazon Japan | 0.38 | 0.06 | -6.9 | ❌ 非マッチ |

---

## まとめ：名寄せ判断のための「3指標の読み方」

```
cosine_similarity  → 意味的な近さ（主指標）
                     0.80以上で「意味が近い」と判断

text_match         → 字面の一致（補助指標）
                     高ければ確信度が増す。
                     低くても cosine が高ければ意味的マッチとして有効

reranker_score     → 最終的な再評価（比較用）
                     同一クエリ内での相対比較が重要。
                     大きな負値は「信頼できないマッチ」のシグナル

_relevance_score   → 3指標の総合値（実際のソートキー）
                     フィルタリングや MATCH_RANK の計算に使用する
```
